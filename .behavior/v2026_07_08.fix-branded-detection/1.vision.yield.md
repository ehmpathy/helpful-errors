# 1.vision — registered-symbol brand + version stamp for HelpfulError

## the wish, restated

let folks detect "is this a HelpfulError?" **globally** — reliably across duplicate/multiple
copies of the `helpful-errors` package and across minification — via a registered symbol brand.
stamp the package version on the brand so callers can assert "this brand is version X or later"
at runtime. expose an ergonomic guard:

```ts
isHelpfulError(error: Error, options?: { version: { min: string } }): error is HelpfulError
```

---

## the outcome world

### day-in-the-life

a downstream package (`with-simple-cache`) throws a `ConstraintError` from a nested copy of
`helpful-errors`. the consuming app has its *own* copy of `helpful-errors` at a different point
in its dependency tree. today, `error instanceof ConstraintError` returns **false** across that
boundary — the two copies define two distinct classes, so the prototype chains never match. the
app's error handler misclassifies the error, logs it as a 500, and pages oncall for what was
actually a 400 caller-fault.

after this wish: the app calls `isHelpfulError(error)` and gets `true` regardless of which copy
minted the error, because the brand lives in the **realm-global symbol registry** (`Symbol.for`),
not on any one module's class. the handler classifies correctly. no page.

### before / after

| scenario | before | after |
|----------|--------|-------|
| same copy, in-process | `instanceof` works | `instanceof` still works; `isHelpfulError` also works |
| duplicate copies (nested deps) | `instanceof` **fails** silently | `isHelpfulError` returns `true` |
| minified bundle (class names mangled) | name-based checks fragile | brand symbol survives minification |
| "is this ≥ v1.8?" at runtime | impossible | `isHelpfulError(err, { version: { min: '1.8.0' } })` |

### the "aha" moment

the value clicks when a consumer realizes they no longer have to reason about *how many copies*
of `helpful-errors` are installed. the brand is a property of the JS realm, not of a module
instance — so "is this ours?" becomes a single global truth instead of a per-copy gamble.

## user experience

### usecases / goals

1. **cross-copy classification** — error-handler middleware (http, cli, queue) asks "is this a
   HelpfulError so i can trust `.code`, `.metadata`, `.redact()`?" without caring about copy identity.
2. **version-gated feature detection** — a consumer that relies on a capability added in a
   specific version (e.g. `.code` semantics) asserts a minimum brand version before using it.
3. **belt-and-suspenders for peer-dep enforcement** — `with-simple-cache` will add
   `helpful-errors` as a peerDependency to force one shared copy (the root-cause fix); the brand
   is the safety net for when a duplicate still sneaks in.

### contract inputs & outputs

```ts
// primary guard — the star of the wish
isHelpfulError(
  error: Error,
  options?: { version: { min: string } },
): error is HelpfulError;

// companion — read the stamped version off any branded error
getHelpfulErrorVersion(error: Error): string | undefined;
```

- `isHelpfulError(err)` → `true` iff `err` carries the registered brand symbol.
- `isHelpfulError(err, { version: { min: '1.8.0' } })` → `true` iff branded **and** the stamped
  version `>=` `'1.8.0'` (semver compare).
- `getHelpfulErrorVersion(err)` → the stamped version string, or `undefined` if unbranded.

### what leveraging it looks like

```ts
import { isHelpfulError } from 'helpful-errors';

// http error middleware
app.use((err, req, res, next) => {
  if (isHelpfulError(err)) {
    return res.status(err.code?.http ?? 500).json(err.toJSON());
  }
  return res.status(500).json({ message: 'internal error' });
});

// version-gated capability use
if (isHelpfulError(err, { version: { min: '1.8.0' } })) {
  // safe to rely on the branded-error contract introduced at 1.8.0
}
```

### timeline

1. import guard → 2. receive unknown `Error` at a boundary → 3. `isHelpfulError(err)` → 4. branch
on the truth → 5. (optionally) read `err.code` / `err.metadata` with confidence.

## mental model

### how a user describes it to a friend

> "it stamps every helpful-error with a global fingerprint, so no matter which copy of the library
> made the error, i can still ask 'is this one of ours?' and get a straight answer."

### analogies

- **passport stamp** — the brand is a stamp in a globally-recognized registry, not a business card
  from one office. any border agent (realm) reads the same stamp.
- **`Symbol.for` as a shared bulletin board** — everyone who asks for the same key gets the same
  pinned note. `Symbol()` would be a private note in your own pocket — invisible to others.

### terms: theirs vs ours

| users say | we say |
|-----------|--------|
| "is it ours / is it a helpful error" | branded / carries the registered brand |
| "which version" | stamped version |
| "duplicate installs" | multiple package copies / realm-global identity |

## evaluation

### how well it solves the goals

- **cross-copy detection**: fully solved. `Symbol.for('...')` is shared across all copies in a
  realm, so the brand check is copy-independent.
- **version gate**: solved via stamping the version as the symbol's value and a semver `>=` compare.
- **minification-proof**: solved. the discriminator is a symbol keyed by a string in the global
  registry — not a class name, not an enumerable property — so bundler mangling can't erase it.

### pros

- narrow, additive surface: two new exports, one internal stamp; no breaking change to prior
  `instanceof` behavior.
- realm-global correctness without a runtime registry we maintain — JS provides the registry.
- non-enumerable brand → invisible to `Object.keys`, `toJSON`, `for...in` (matches the extant
  `original` property discipline in `HelpfulError.ts:111`).

### cons / costs

- a registered symbol is a (small) permanent global-registry entry keyed by string; the key must
  be collision-safe and stable forever (renaming it later is a breaking change).
- version compare pulls in semver logic. we need a tiny, dependency-light compare (or reuse a
  scoped util) to honor the no-barrel / cycle-safe import discipline.
- the brand value must reflect the *minting copy's* version — so each copy stamps its own version.
  a `min` check therefore asserts "the copy that made this error was ≥ X", which is exactly the
  intended semantic, but worth stating explicitly.

### edgecases & pit-of-success

| edgecase | behavior | keeps user safe by |
|----------|----------|--------------------|
| plain `Error` (not ours) | `isHelpfulError` → `false` | never false-positive on foreign errors |
| non-Error input (`null`, string) | `false` (guard tolerates) | boundary-safe; no throw |
| branded error, no version arg | `true` | simplest path is the common path |
| `min` newer than stamped version | `false` | honest "this copy predates the capability" |
| duplicate copy, older version | `isHelpfulError` `true`, version gate may be `false` | detection still works; gate is precise |
| malformed stamped version | treat as fails-gate / `undefined` version | never crash on a bad stamp |

## open questions & assumptions

### assumptions

- **A2** `[pivoted 2026-07-17]`: the brand is a **static** property on the base `HelpfulError` class
  (`public static readonly [MARK_AS_HELPFUL_ERROR] = version`), so *all* subclasses
  (`BadRequestError`, `ConstraintError`, `MalfunctionError`, `UnexpectedCodePathError`) inherit it
  automatically via the constructor's prototype chain — **verified** all chain to `HelpfulError`
  (`BadRequestError.ts:15`, `UnexpectedCodePathError.ts:8`, `MalfunctionError.ts:16`,
  `ConstraintError.ts:16`). detection reads the brand off the value's *class*
  (`value.constructor[MARK]`), so `redact()` — which re-constructs — keeps redacted clones branded
  for free, at zero per-construction cost. this mirrors domain-objects' static-brand shape. the
  original plan stamped the brand per-instance in the constructor via `Object.defineProperty`; the
  static form supersedes it (less work per throw, no instance property to hide from `toJSON`).
  **immutability preserved (deliberate)**: the static brand is defined via `Object.defineProperty`
  with `writable:false, enumerable:false, configurable:false` — NOT a plain `static readonly` class
  field. a plain field is, at runtime, writable+enumerable+configurable (the TS `readonly` is
  compile-time only), which would let any caller overwrite or `delete` the brand and silently defeat
  detection. `Object.defineProperty` keeps the exact tamper-proof, hidden guarantee the per-instance
  stamp had (and that the `original` property uses), so the pivot loses no safety.
- **A3**: version semantics for `min` are semver `>=` (from the wish phrase "a given version or
  later"). note: comparison **must** be semver, not lexicographic — `'1.10.0' < '1.7.3'` as strings
  is a latent bug.

### questions — triaged

each tagged `[answered]` (settled now), `[research]` (research phase), or `[wisher]` (needs wisher).

- **Q1 (version source)** `[answered — pivoted 2026-07-17]` — the package has **no** runtime handle
  on its own version today (verified: no `version.ts`). the original decision was a build-step
  codegen that writes `src/version.ts` before `build:compile`, out of a fear that a `../package.json`
  import escapes `rootDir: src` and breaks the flat `dist` layout. **that fear was unfounded**:
  domain-objects ships the identical `rootDir: src` layout yet imports `../../package.json` directly
  and still emits a flat `dist`. **pivot decision (wisher, 2026-07-17)**: drop the codegen entirely;
  `src/version.ts` now does `import pkg from '../package.json'; export const version = pkg.version;`
  (via `resolveJsonModule`), a mirror of domain-objects' `src/instantiation/version.ts`. proven:
  `npm run build` emits a flat `dist/version.js` whose `require('../package.json')` resolves this
  copy's own manifest at runtime. no build step, no `preversion` hook, no top-level execution risk.
- **Q2 (`getHelpfulErrorVersion` public?)** `[answered — wisher]` — **yes, expose it publicly**
  (wisher, 2026-07-10). symmetric with `isHelpfulError`; both land in `src/index.ts`.
- **Q3 (brand value shape)** `[answered]` — raw version **string** as the symbol value, per the
  wish example `{ value: version }`. no wrapper object; matches the wish, avoids speculative surface
  (wet-over-dry).
- **Q4 (`min` only, or `max`/exact?)** `[answered]` — `min` only. the wish asks only for "version or
  later"; extra comparators are speculative. add later if a real consumer needs them.
- **Q5 (registry key namespace)** `[answered — wisher]` — **flat `'helpful-errors'`** (wisher,
  2026-07-10). the registered-symbol key is `Symbol.for('helpful-errors')`, per the wish. this is a
  permanent public contract.
- **Q6 (input width)** `[answered]` — the runtime guard accepts `unknown` and narrows to
  `error is HelpfulError` (standard type-guard shape), so callers who `catch (e: unknown)` need no
  pre-cast; non-Error / null inputs return `false` without throw.
- **Q7 (scope of "global")** `[answered]` — "global" means **in-process, realm-global**. the brand
  does **not** survive JSON-serialize/revive or cross-realm transfer (`worker_threads`, `vm`), since
  those have separate symbol registries and the revived value is no longer the live object. this
  matches the named `with-simple-cache` blocker (same process, duplicate installs); the text in this
  vision now qualifies "global" as in-process to avoid a false expectation.

### what to research externally

- **`[research]`** confirm a lightweight semver-compare approach that respects the repo's import
  discipline (no `type-fns` barrel; scoped imports only — see

## groundwork

### external research

- **`Symbol.for` / global symbol registry** — verified against well-established JS semantics
  (ECMAScript `Symbol.for(key)` returns the same symbol for a given key across the entire realm's
  global symbol registry; `Symbol()` always mints a unique, module-local symbol). the wish's
  hardest-flagged pitfall — "must be `Symbol.for()`, never `Symbol()`" — is **correct**: a plain
  `Symbol()` is per-module, so each duplicate copy would mint a different one and the brand would be
  invisible across copies. no live API/service dependency to verify beyond the language spec.

### internal research

- **base class + inheritance** (`src/HelpfulError.ts`): `HelpfulError extends Error`
  (`HelpfulError.ts:56`). the constructor already sets a **non-enumerable** property via
  `Object.defineProperty(this, 'original', { enumerable: false, ... })` (`HelpfulError.ts:111`) —
  this is the exact pattern the brand should follow, so the brand stays out of `toJSON`
  (`HelpfulError.ts:292`), `Object.keys`, and `for...in`.
- **subclass chain** confirmed all route to `HelpfulError`:
    - `ConstraintError extends BadRequestError` (`src/ConstraintError.ts:16`)
    - `BadRequestError` → (verify extends HelpfulError; peers `MalfunctionError`,
      `UnexpectedCodePathError` similarly) — all rely on `super(...)`, so a brand stamped in the base
      constructor propagates to every subclass with zero per-subclass work.
- **public surface** (`src/index.ts`): currently exports `BadRequestError`, `ConstraintError`,
  `getError`, `HelpfulError` (+ `HelpfulErrorCode` type), `MalfunctionError`,
  `UnexpectedCodePathError`, `withHelpfulError`. new exports `isHelpfulError` /
  `getHelpfulErrorVersion` slot in here.
- **import discipline**: repo forbids barrel imports from `type-fns` (cycle hazard —
  `.agent/repo=.this/role=any/briefs/rule.forbid.barrel-imports-from-typefns.md`); `HelpfulError.ts`
  already uses scoped imports (`type-fns/dist/checks/isPresent.assess`, `.../companions/omit`). any
  new util import must follow the scoped pattern.
- **naming/vocab**: guard follows the `is*` transformer convention already used across ehmpathy
  (`is` prefix, `error is HelpfulError` type predicate). matches repo name rules.
- **no runtime version access today** — grep for `version` in tsconfig/src found no mechanism;
  this confirms Q1 is genuinely unresolved (not something already solved elsewhere).

## what is awkward?

- **version sourcing (Q1)** is the forced-feeling part. the wish treats `version` as if it's just
  available (`{ value: version }`), but the package has no runtime handle on its own version. every
  clean-looking option (import package.json, codegen a const, build-time inject) carries a
  build-layout or tooling cost. this is where the design fights reality and must be settled first.
- **"stamp each copy's own version" nuance**: a `min` check answers "was the *minting* copy ≥ X?",
  not "is the *consuming* copy ≥ X?". this is the correct and only feasible semantic, but it's a
  subtle mental-model bump — a user might expect the check to reflect their own installed version.
- **permanent global key**: committing to `Symbol.for('helpful-errors')` forever is a mild
  discomfort — it's an un-versioned, un-namespaced global string. collision risk is low but the key
  can never change without breaking cross-version detection (which is the whole point).
- **semver dependency tension**: wanting a `min` compare while honoring strict no-barrel/cycle-safe
  imports nudges us toward hand-rolling a tiny compare rather than pulling a dependency — slightly
  awkward but aligns with the repo's minimal-dependency posture.
