### why no integration/acceptance suites

per `rule.require.test-coverage-by-grain`: `helpful-errors` is a pure library â€” the new operations
are **public transformers** (`isHelpfulError`, `getHelpfulErrorVersion`, `isVersionAtLeast`) plus a
constructor stamp. they are exported from `src/index.ts` (public surface) yet are **transformer-grain**:
pure, deterministic, no remote boundary crossed. so **unit** is the correct and complete grain â€” there
is no cli/api/service contract that would warrant an integration or acceptance suite.

the snapshot matrices below are therefore an **added vibecheck aid** on top of the required unit grain
(so reviewers can eyeball the full truth table), **not** an acceptance-grain obligation. "transformer"
(the grain) and "public export" (the surface) are both true and non-contradictory: the grain drives the
required test scope (unit), the surface motivates the extra snapshot courtesy.

#### why NOT a real dual-install integration test

a reviewer may ask for the "realest" cross-copy proof: literally install two copies of
`helpful-errors` (separate `node_modules` trees) and assert `isHelpfulError` detects across them.
this is **deliberately out of scope**, on three grounds:

1. **vision Q7 [answered]** scopes "global" as **in-process, realm-global**. two copies in one node
   process share **one** realm symbol registry, so a dual-install harness exercises the *same*
   `Symbol.for('helpful-errors')` lookup the `given a foreign copy` unit test already exercises. the
   only delta a separate tree adds is a distinct **class** object â€” and detection is class-independent
   by design (the test proves `foreignCopy instanceof HelpfulError === false` while
   `isHelpfulError(foreignCopy) === true`).
2. **grain rule** (`rule.require.test-coverage-by-grain`): these are pure transformers â†’ unit grain. a
   dual-install harness is an integration test â€” the wrong grain; to add it would break the discipline.
3. **already human-overruled**: this exact request (a build-and-install-twice cross-copy integration
   test) was raised on stone 5.1 and the human explicitly overruled it as a goalpost-move against the
   vision's unit-grain choice (see `blocker/5.1.execution.from_vision.md`).

`Symbol.for(key)` that returns one shared symbol across separate `node_modules` trees in a single
realm is an **ECMAScript/V8 language guarantee**, not app behavior this package can regress â€” so a test
of it would assert the language spec, not our code (low marginal value, per `rule.prefer.wet-over-dry`).
